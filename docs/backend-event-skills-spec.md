# 后端事件触发 Skills 机制

## 目标

把机器人从“展示卡片”改成后端业务事件机制。前端只提交业务动作，后端根据事件类型自动触发对应机器人和 Skills，并把结果写回业务对象、事项中心和事件日志。

## 一期范围

| 业务动作 | 后端事件 | 触发机器人 | Skills |
|---|---|---|---|
| 上传发票 | `invoice.uploaded` | 报销助理 | 发票识别与命名、报销规则校验、项目费用归类 |
| 上传合同/邮件材料 | `contract.uploaded` | 合同审批助理 | 合同材料读取、合同风险分析、合同项目组创建、审批链路路由、履约风险监控 |
| 创建会议/预订会议室 | `meeting.created` | 日程和会议助理 | 日程会议协调、会议纪要提取、访客接待引导 |
| 提交日报 | `daily_report.submitted` | 日报周报助理 | 日报周报汇总、工作量化评分、会议纪要提取 |

## 非目标

- 一期不做企微回填。
- 一期不做“运行机器人”按钮。
- 一期不把后台 Skill 明细暴露给普通员工。

## 验收标准

- 每个业务动作都有明确后端接口。
- 每个接口成功后都会生成 `automationEvent`。
- `automationEvent` 包含：事件类型、业务动作、机器人、Skills、输出对象、任务 ID、触发人、时间。
- Skills 由后端 `skillRegistry` 管理，包含版本、提示词/规则、输入输出结构、写回对象和更新时间。
- Skill 上传入口是通用文件入口，但执行函数按 Skill ID 分开，例如 `invoice-ocr`、`contract-risk`、`workload-score` 都有独立 handler。
- 老板和总助可通过 `POST /api/skills/:id/upload` 上传 `.json/.md/.txt` 替换 Skill，新事件会使用新版本。
- 每个 Skill 在机器人编排视图都有独立“替换入口”，上传后只替换当前 Skill，不影响其他 Skill。
- 老板和总助可在机器人编排视图看到后端触发规则和最近事件。
- 普通员工只看到业务结果和待办，不直接操作 Skills。
